> 最后更新：2026-04-22

## 一、开发流程

所有功能开发必须严格按以下流程执行，不可跳步：

```
1. 讨论     ← 理解需求，明确范围
      ↓
2. 方案文档 ← 出规划文档，评估数据源、字段映射、风险
      ↓
3. 用户确认 ← 用户审核方案，确认后方可实施
      ↓
4. 实现     ← 编码实现（派虾1或自己改）
      ↓
5. 验证     ← 测试、数据校验
      ↓
6. Commit   ← 代码 + 文档一起提交
```

**禁止事项：**
- ❌ 不出方案文档直接写代码
- ❌ 用户未确认直接执行
- ❌ 方案文档不随代码 commit
- ❌ "review" 变成 "review + 修复"

## 二、数据写入铁律

### 2.1 upsert 字段保护

**铁律：值为 None 的字段不得覆盖数据库已有值。**

```python
# ❌ 错误：None 会覆盖已有值
ON CONFLICT (stock_code, trade_date) DO UPDATE SET market_cap = EXCLUDED.market_cap

# ✅ 正确：只更新非 None 字段
ON CONFLICT (stock_code, trade_date) DO UPDATE SET
  market_cap = CASE WHEN EXCLUDED.market_cap IS NOT NULL THEN EXCLUDED.market_cap ELSE daily_quote.market_cap END,
  pe_ttm = CASE WHEN EXCLUDED.pe_ttm IS NOT NULL THEN EXCLUDED.pe_ttm ELSE daily_quote.pe_ttm END
```

`db.py` 的 `upsert` 函数已内置此逻辑（COALESCE 保护），调用方无需手动处理。如需显式将字段置为 NULL，使用 `force_null_cols` 参数。

### 2.2 历史回填 vs 实时同步

当同一张表存在多种写入方式时，必须在方案文档中明确：

| 问题 | 必须回答 |
|------|---------|
| 各写入方式提供哪些字段？ | 列出字段矩阵 |
| 是否有字段冲突？ | 标注哪些字段会互相覆盖 |
| 如何保护已有数据？ | 说明保护机制 |
| 回填是否影响实时数据？ | 说明影响范围 |

### 2.3 物化视图依赖

物化视图依赖基础表的数据完整性。写入基础表后是否需要刷新物化视图、刷新顺序是什么，必须在方案文档中说明。

## 三、新功能开发检查清单

接入新数据源或开发新功能前，必须逐项确认：

### 3.1 数据源评估

- [ ] 数据源稳定性如何？是否可能被封 IP / 限流？
- [ ] 提供哪些字段？不提供哪些字段？
- [ ] 与现有数据源是否有数据重叠？如何处理？
- [ ] 是否需要代理？哪个环境需要？

### 3.2 字段与表设计

- [ ] 是否需要新建表？表结构是什么？
- [ ] 是否需要修改现有表？（DDL 变更需更新 SCHEMA.md）
- [ ] upsert 冲突键是什么？
- [ ] 哪些字段会因 None 覆盖而丢失？

### 3.3 同步策略

- [ ] 全量同步还是增量同步？增量判断依据是什么？
- [ ] 限流策略是什么？间隔多少秒？
- [ ] 重试策略是什么？最大重试次数？
- [ ] 预计耗时多少？是否需要 tmux 后台运行？
- [ ] 边拉边写还是拉完再写？commit 频率？
- [ ] 断点续传如何实现？

### 3.4 风险评估

- [ ] 对现有数据有什么影响？
- [ ] 失败时如何回滚？
- [ ] 是否需要物化视图刷新？

## 四、文档规范

### 4.1 方案文档结构

每个功能方案文档必须包含以下章节：

```markdown
# 功能名称

## 背景
## 数据源评估
## 字段映射
## 表结构设计（如需新建）
## 实现方案
## 同步策略（限流、重试、断点续传）
## 风险评估
## 验证方案
```

### 4.2 现有文档维护

| 文档 | 何时更新 | 更新内容 |
|------|---------|---------|
| SCHEMA.md | 新建/修改表 | DDL 变更 |
| ARCHITECTURE.md | 架构变更、新增数据源 | 设计决策、数据源矩阵 |
| ROADMAP.md | 功能完成/新增 | 进度标记 |
| SPEC.md | 数据量变化、功能变更 | 数据现状、已完成/待办 |
| DEV_GUIDELINES.md | 规范变更 | 新增规则 |

## 五、数据库变更规范

### 5.1 DDL 变更

- 所有建表/改表 SQL 必须记录在 `scripts/` 目录
- `SCHEMA.md` 必须同步更新
- 变更必须向后兼容（不删除已有列、不改已有列类型）

### 5.2 物化视图

- 新建物化视图必须在方案文档中说明依赖关系
- 刷新顺序必须在文档中明确
- 刷新 SQL 必须收录到 `scripts/materialized_views.sql`

### 5.3 生产环境

- schema 变更需要用户确认后执行
- 大批量数据操作（全量回填等）用 tmux 后台运行，不用 nohup
- 操作前确认磁盘空间充足

## 六、代码规范

### 6.1 限流

| 场景 | 最小间隔 | 推荐间隔 |
|------|---------|---------|
| 公开 API（腾讯、东方财富） | 1s | 2-5s |
| SEC EDGAR | 0.1s | 0.5s |
| akshare 内部调用 | 0.3s | 0.5s |
| 大批量回填 | 2s | 3-8s（随机抖动） |

### 6.2 错误处理

- 不吞异常。所有外部调用必须 catch 并记录上下文（stock_code、参数、响应状态）。
- 失败重试使用指数退避（1s → 2s → 4s → 8s），最大重试 3 次。
- 连续失败超过阈值触发熔断，暂停该数据源的后续请求。

### 6.3 日志

| 级别 | 使用场景 |
|------|---------|
| INFO | 每只股票拉取完成、批量写入完成、同步进度 |
| WARNING | 单只失败、限流触发、熔断触发 |
| ERROR | 数据解析失败、数据库写入失败 |

关键操作必须打日志：
- 每只股票的拉取结果（成功/失败/耗时）
- 每批写入的行数
- 同步总进度（X/Y、预计剩余时间）

### 6.4 测试

- 新增 fetcher 函数必须用单只股票做验证测试（不实际跑全量）
- 测试脚本临时文件（如 `test_*.py`）不要入库，验证完删除
- 正式测试用例放在 `tests/` 目录

### 6.5 sync/ 包规范

`sync/` 包从 `sync.py`（1751 行）拆分而来，CLI 入口为 `python -m sync`。

**模块职责：**

| 模块 | 职责 | 谁调用 |
|------|------|--------|
| `_utils.py` | 日志、`ensure_sync_progress_table`、`MARKET_CONFIG`、`sync_one_stock` | 其他模块内部使用 |
| `manager.py` | `SyncManager` 类，调度所有同步任务 | `__init__.py`、`scheduler.py` |
| `daily_quote.py` | 腾讯 K 线历史日线回填 | `__init__.py` CLI |
| `share.py` | 股本数据同步 | `__init__.py` CLI |
| `us_market.py` | 美股 SEC EDGAR 同步 + 重解析 | `__init__.py` CLI、`scheduler.py` |
| `__init__.py` | CLI `main()` + 对外接口导出 | 直接调用、`python -m sync` |
| `__main__.py` | `python -m sync` 支持 | 用户 CLI |

**新增同步任务规则：**

1. 独立的同步函数放在新文件（如 `sync/my_task.py`）
2. SyncManager 方法（依赖 `self.force`）留在 `manager.py`
3. CLI 参数在 `__init__.py` 的 `main()` 中注册
4. 外部导入通过 `__init__.py` 的 `__all__` 导出（如 `from sync import sync_us_market`）
5. `scheduler.py` 通过 `from sync import SyncManager` 使用

## 七、教训记录

以下是从实际开发中总结的教训，避免重复踩坑：

| # | 教训 | 根因 | 预防措施 |
|---|------|------|---------|
| 1 | upsert 无 None 保护导致市值数据丢失 | `db.py` upsert 全量覆盖 | upsert 已实现 COALESCE 保护（`force_null_cols` 可显式覆盖） |
| 2 | TTM 计算 annual+quarterly 混合导致数值虚高 3 倍 | 窗口函数不区分报告类型 | 只用 annual 计算 TTM |
| 3 | 历史回填覆盖实时行情字段 | 方案文档未评估字段覆盖风险 | 新功能开发必须检查字段矩阵 |
| 4 | 方案文档缺失导致设计缺陷 | 缺少系统级规范 | 强制先文档后开发 |
